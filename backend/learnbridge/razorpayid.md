# Razorpay Transaction ID Integration Guide

This document explains the issue with original Razorpay IDs disappearing and what was precisely changed to resolve it, ensuring that you see the real Razorpay Transaction ID in the Teacher Wallet when an Admin hits the transfer button.

## 1. What was the Issue?
Whenever a student purchased a Course or a Live Class, the original `pay_...` Razorpay ID was correctly generated by the Razorpay gateway. 
However, when the Admin opened the Admin Wallet page and clicked the **"Transfer"** button to send the funds to the teacher, the original Razorpay ID was overridden in the database and replaced with an auto-generated transaction string like `TRX-87-1772559175`. 
Additionally, for **Live Classes**, the Razorpay ID wasn't even being mapped during the initial payment registration step.

## 2. What Changes Were Made?

### A. Live Class Payment Forwarding (`backend/LearnBridge/liveclass/views.py`)
In `VerifyLiveClassPaymentView` inside the Live Classes app, the student successfully verifies their Razorpay payment. Previously, this Razorpay ID wasn't passed down to the wallet function:
```python
# Added razorpay_payment_id parameter to the wallet crediting function
credit_live_class_wallet(
    live_class=live_class,
    amount=live_class.registration_fee,
    description=f"Live Class Registration - {live_class.title}",
    student=request.user,
    razorpay_payment_id=razorpay_payment_id   # <-- THIS WAS ADDED
)
```

### B. Wallet Service Modifications (`backend/LearnBridge/wallet/services.py`)
Because we started sending the Razorpay payment explicitly from the Live Class views, we also had to update the actual wallet service definition (`credit_live_class_wallet`) to accept the ID and save it into the `TeacherTransaction` pending table row:
```python
def credit_live_class_wallet(live_class, amount, description, student, razorpay_payment_id=None):
    # ...
    # This automatically associates the original Razorpay string
    # with the Teacher's pending transaction from the moment it is bought!
    TeacherTransaction.objects.create(
        teacher_wallet=teacher_wallet,
        live_class=live_class,
        student=student,
        course=None,
        source="live_class",
        amount=teacher_share,
        description=description,
        status="payment_pending",
        transaction_id=razorpay_payment_id  # <-- THIS WAS ADDED
    )
```
*(Note: Course payments were already doing this correctly in `credit_admin_wallet` and `payments/views.py`, so only `live_class` logic needed updating).*

### C. Admin Transfer View Optimization (`backend/LearnBridge/wallet/views.py`)
The most important fix resided in `AdminTransferToTeacherView`. When the Admin clicked *Transfer*, the backend updated the `teacher_tx.status` to `"payment_completed"`, but completely wiped out the original ID.

**Old Code:**
```python
teacher_tx.transaction_id = (
    admin_tx.razorpay_payout_id
    if admin_tx.razorpay_payout_id
    else f"TRX-{admin_tx.id}-{int(timezone.now().timestamp())}"
)
```

**New Code:**
```python
if not teacher_tx.transaction_id:
    teacher_tx.transaction_id = (
        admin_tx.razorpay_payout_id
        if admin_tx.razorpay_payout_id
        else f"TRX-{admin_tx.id}-{int(timezone.now().timestamp())}"
    )
```

By wrapping the logic in `if not teacher_tx.transaction_id:`, we tell Django: 
**"If a transaction ID already exists (because the Razorpay ID safely stored it during the purchase), DO NOT overwrite it. Only assign a `TRX...` fallback if it is totally empty."**

## 3. How It Works Now
1. **Student Buy:** Student pays for a Live Class or Course.
2. **Payment Tracking:** The generated Razorpay `pay_xxx...` ID is sent directly to `TeacherTransaction` table as a pending payment (`transaction_id`).
3. **Admin Transfer:** Admin clicks "Transfer". 
4. **ID Preservation:** The system checks if `TeacherTransaction` already houses an ID. Because it holds `pay_xxx...`, it skips creating a new `TRX...` code. The backend marks the status as completed and releases the exact Razorpay ID back to the teacher frontend!
